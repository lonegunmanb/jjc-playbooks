# Bug 类 Issue 处理指南（面向维护者）

## 1. 适用范围

当 Issue 被归类为 Bug（预期行为未被满足且可复现）时，Agent 必须使用本指南协助维护者完成预处理。

本阶段目标是：

1. 帮维护者快速判断问题是否真实存在与是否可复现。
2. 在人工审批前，产出高质量的根因分析与修复计划。
3. 默认优先设计非破坏性修复路径，降低升级成本与回归风险。

## 2. 处理原则（强制）

1. 先复现，后结论
   - 没有复现证据时，不做确定性技术结论。
2. 先最小化影响，后扩展优化
   - 修复目标先满足正确性与兼容性，再考虑重构或增强。
3. 非破坏性优先
   - 修复方案应尽可能保证：
   - 现有用户仅需升级模块版本即可。
   - 对既有稳定配置，terraform plan 不应出现 drift。
   - 不应引入意外 update/add/delete。
4. 计划先行，审批后执行
   - Agent 在该阶段主要产出可审批计划与证据；涉及实现、合并、发布的动作需维护者审批。
5. 安全优先
   - 若分析中发现潜在安全风险，立即按安全流程升级处理并停止在公开 Issue 展开细节。

## 3. 快速处理流程

对每个 Bug Issue，按顺序执行：

1. 问题归一化
   - 提炼一句话问题定义：预期行为、实际行为、影响范围。
2. 证据收集
   - 读取 Issue 正文、评论、标签、相关版本信息。
   - 检查仓库 examples、变量约束、模块实现和最近变更。
3. 重复与回归判断
   - 检查是否与 open/closed Issue 或近期 PR 重复。
   - 判断是否可能是回归（最近版本引入）。
4. 本地复现（强烈建议）
   - 优先基于 examples/ 构造最小复现用例。
   - 记录可复现性、前提条件、失败日志要点。
   - 复现尝试结束后执行环境清理（优先 terraform destroy）。
5. 根因分析
   - 定位到变量约束、条件分支、资源参数映射、依赖顺序、provider 行为或文档误导中的一类或多类。
6. 修复路径设计
   - 至少给出一个默认修复方案（唯一推荐）。
   - 明确兼容性影响、测试策略、回滚策略。
7. 修复验证
   - 基于修复路径尝试实现并验证是否解决问题。
   - 若单次尝试失败，调整方案后重试，最多尝试 10 次。
   - 记录每次尝试的方案摘要与失败原因。
8. 维护者简报输出
   - 使用第 4 节模板输出可直接决策的信息。

## 4. Agent 输出模板（给维护者）

对每个 Bug Issue，Agent 输出必须包含：

1. 分类摘要
   - 建议类别：Bug
   - 置信度：高/中/低
   - 证据摘录：来自 Issue 的关键原文
2. 复现结论
   - 复现状态：可复现/不可复现/部分可复现
   - 复现环境：Terraform 版本、Provider 版本、模块版本、关键前置条件
   - 最小复现路径：基于哪个 example 改造，改了哪些关键参数
3. 根因分析
   - 根因类型：逻辑错误/默认值问题/条件判断缺陷/依赖顺序/provider 差异/文档歧义/其他
   - 证据：代码位置、行为对比、日志证据
4. 修复计划
   - 验证状态：已验证可修复/修复方案未知
   - （已验证时）默认方案（必须唯一）：
     - 修复目标：
     - 拟修改范围：
     - 兼容性评估：非破坏性/可能破坏性
     - 测试计划：单测、示例验证、回归验证
     - 风险与缓解：
   - （修复方案未知时）已尝试方案汇总：
     - 每次尝试的方案摘要、失败现象与原因
     - 已排除的技术路径
     - 剩余可探索方向（如有）
5. 维护者行动建议（审批项）
   - 建议决策：批准实现/要求补充信息/转其他类型
   - 负责人：
   - 时间框：
   - 完成标准：
6. 可直接发送的回复草稿
   - 面向 Issue 提交者的简洁回复，说明当前结论与下一步。

## 5. 本地复现指南（Agent 执行）

### 5.1 复现策略

1. 先选择最接近场景的现有 example。
2. 若无匹配示例，创建最小化临时复现配置（仅保留触发 bug 的必要资源与参数）。
3. 固定关键版本（Terraform、Provider、模块）以避免噪声。
4. 明确输入变量与前提假设（区域、SKU、权限、网络条件）。

### 5.2 复现记录最小集合

Agent 必须记录：

1. 复现配置摘要（不包含敏感信息）。
2. 触发步骤摘要。
3. 预期行为与实际行为对比。
4. 关键报错或差异输出摘要。
5. 复现是否稳定（每次都出现/偶发）。

### 5.3 不可复现时的处理

当无法复现时，默认行动方案为：

1. 列出缺失信息清单（输入参数、版本、日志片段、是否使用私有网络等）。
2. 生成一次性补充信息模板回复提交者。
3. 标记为 Needs: Author Feedback，等待作者补充。

### 5.4 复现后的清理要求（强制）

每次复现尝试完成后，Agent 应尽力清理测试资源，避免成本与配额残留。

1. 首选：运行 terraform destroy
   - 在复现目录执行 terraform destroy，确保本次测试创建的资源被销毁。
2. 兜底：若 destroy 超时或报错，触发资源组异步删除
   - 可执行命令：az group delete --name <resource-group-name> --no-wait --yes
   - 说明：该命令用于异步触发删除，不要求同步等待删除完成。
3. 记录清理结果
   - 在维护者简报中记录：是否 destroy 成功、是否触发了资源组异步删除、对应资源组名称。

## 6. 非破坏性修复约束（核心）

### 6.1 默认验收标准

修复方案默认必须满足以下标准（除非维护者批准例外）：

1. 升级路径简单
   - 已使用最新模块的用户仅升级版本号即可迁移。
2. 计划稳定
   - 对既有稳定配置执行 terraform plan，应无 drift。
3. 行为兼容
   - 不改变现有输入的语义，不引入强制新参数。
4. 状态兼容
   - 不要求用户手工改 state，不要求 import/move 作为默认路径。

### 6.2 若无法避免破坏性变更（必须高亮）

若技术上无法避免破坏性变更，Agent 必须在维护者简报中使用醒目标记：

[BREAKING-CHANGE][REQUIRES-MAINTAINER-APPROVAL]

并且必须额外提供：

1. 为什么无法非破坏性修复（技术约束与替代方案比较）。
2. 受影响用户画像与影响范围估计。
3. 迁移指南草案（最少步骤、可回滚）。
4. 版本策略建议（是否需要 major 版本变更）。
5. 风险提示：terraform plan 可能出现的 update/add/delete 类型。

未提供以上内容时，不允许建议进入实现阶段。

## 7. 修复验证指南（Agent 执行）

### 7.1 验证策略

1. 基于第 3 节步骤 6 设计的默认修复方案，在复现环境中实际实施修复。
2. 实施后重新执行复现用例，验证问题是否已消除。
3. 同时验证非破坏性约束（对既有稳定配置 terraform plan 无意外 drift）。

### 7.2 失败重试机制

1. 若验证失败，分析失败原因，调整修复方案后重试。
2. 每次重试必须基于上次失败原因做出有意义的方案调整，禁止无变化重复尝试。
3. 最多尝试 10 次。超过 10 次仍未修复，终止验证并进入"修复方案未知"流程。

### 7.3 验证记录（每次尝试必须记录）

| 字段 | 说明 |
|---|---|
| 尝试编号 | 从 1 开始递增 |
| 方案摘要 | 本次尝试的修复思路与具体改动（与上次的差异） |
| 验证结果 | 成功/失败 |
| 失败原因 | 失败时填写：现象、报错、与预期的差距 |
| 下一步调整方向 | 失败时填写：基于本次失败拟做的调整 |

### 7.4 验证成功时的处理

1. 记录最终成功方案与验证证据。
2. 将验证结果纳入维护者简报的修复计划部分，标记验证状态为"已验证可修复"。
3. 执行环境清理（同第 5.4 节）。

### 7.5 修复方案未知时的处理（10 次尝试均失败）

当 10 次尝试均失败时：

1. 汇总所有尝试的方案摘要与失败原因，形成排除记录。
2. 在维护者简报中标记验证状态为"修复方案未知"。
3. 提供已排除的技术路径与剩余可能方向（如有）。
4. 建议维护者决策：是否升级处理、引入外部专家、或挂起等待上游修复。
5. 执行环境清理（同第 5.4 节）。

## 8. 维护者决策路由（Bug 内部）

| 路由 | 触发条件 | Agent 提供内容 | 建议标签/状态 | 默认行动方案 |
|---|---|---|---|---|
| B1: 可复现且可非破坏修复 | 已稳定复现，根因明确，存在兼容修复路径，且已验证可修复 | 复现证据 + 根因 + 已验证的修复计划 | 保留 Type: Bug，移除 Needs: Triage | 负责人：模块维护者。期限：2 个工作日内批准实现计划。完成标准：实现计划获批并进入开发。 |
| B2: 可复现但可能破坏性 | 已复现且根因明确，但无法保证兼容 | 破坏性影响评估 + 迁移草案 + 版本建议 | Type: Bug + Needs: Attention | 负责人：模块维护者/版本负责人。期限：1 个工作日内完成破坏性审查。完成标准：明确是否接受破坏性方案。 |
| B3: 证据不足待补充 | 当前不可稳定复现或缺关键上下文 | 缺失信息清单 + 一次性追问模板 | Needs: Author Feedback | 负责人：值班维护者。期限：1 个工作日内追问。完成标准：作者补齐信息或进入超时策略。 |
| B4: 实为其他类型 | 分析后发现不属于 Bug（文档/功能/安全/问题反馈） | 重分类证据与目标类别理由 | 调整为对应类型标签 | 负责人：值班维护者。期限：同日。完成标准：类别完成重定向并给出解释回复。 |
| B5: 可复现但修复方案未知 | 已复现且根因明确，经 10 次尝试仍无有效修复方案 | 复现证据 + 根因 + 已尝试方案汇总与失败原因 + 剩余可探索方向 | Type: Bug + Needs: Attention | 负责人：模块维护者/技术负责人。期限：3 个工作日内评估升级路径。完成标准：确定替代方案或升级处理方式。 |

当前结论：
1. 受影响条件：<简述>
2. 预期行为：<简述>
3. 实际行为：<简述>

### 9.2 不可复现（需要补充信息）

感谢反馈。我们已尝试基于现有信息复现，但目前仍缺少关键上下文，暂时无法稳定重现。

1. Terraform 版本与 Provider 版本
2. 模块版本与关键输入参数（可脱敏）

### 9.3 可能涉及破坏性变更

目前可行修复路径可能包含行为变更，存在对现有用户配置造成影响的风险。我们正在向维护者提交破坏性评估与迁移方案，待审批后再进入实现，避免对现网用户造成意外影响。

## 10. 计划审批门槛（Gate）

在进入代码实现前，Agent 提交的计划至少要满足：

1. 有复现证据或有明确的不可复现原因与补充请求。
2. 有根因假设且与证据一致。
3. 有唯一默认修复方案，或已明确记录修复方案未知及全部尝试记录。
4. 有兼容性评估与测试计划。
5. 若涉及破坏性变更，已提供第 6.2 节全部材料并明确高亮。

## 11. 维护者 KPI（可选但推荐）

按月跟踪：

1. Bug Issue 首次响应时长。
2. 可在 3 个工作日内完成复现结论的比例。
3. 非破坏性修复占比。
4. Bug 修复后 reopen 比例。
5. 因破坏性影响导致的升级阻塞 Issue 数量。
