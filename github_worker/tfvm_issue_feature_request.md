# 功能请求类 Issue 处理指南（TFVM，面向维护者）

## 1. 适用范围

当 Issue 被归类为 `功能请求`（请求新功能或增强现有功能）时，Agent 必须使用本指南协助维护者完成预处理。

本阶段目标是：

1. 判断请求的功能是否在模块职责范围内且技术上可行。
2. 评估实现该功能对现有用户的兼容性影响。
3. 产出高质量的可行性分析与实现计划，供维护者决策。

## 2. 处理原则（强制）

1. 先验证可行性，后承诺实现
	- 在确认 AzureRM Provider 支持、Azure API GA 状态和模块架构兼容之前，不做实现承诺。
2. 向后兼容优先
	- 新功能必须以可选方式引入，不改变现有输入的语义或默认行为。
	- 新增变量必须有合理默认值，确保现有用户升级时 terraform plan 无 drift。
3. 范围边界清晰
	- 明确区分"模块应做"和"用户应在模块外自行处理"的边界。
	- 不把 Provider 层面的限制或 Azure 平台层面的限制误判为模块缺失功能。
4. 计划先行，审批后执行
	- Agent 在该阶段主要产出可行性分析与实现计划；涉及代码实现、合并、发布的动作需维护者审批。
5. 安全优先
	- 若功能请求涉及安全敏感操作（如密钥管理、权限变更），需额外评估安全影响。
6. 仅支持 GA（正式发布）功能
	- 我们不支持 Azure Preview 功能。如果请求的功能仅存在于 Azure REST API 的 Preview 版本中，而 GA（正式）版本中不包含，则该功能属于 Preview，不予实现。
	- 判定方法：查阅 Azure REST API 规范（`github.com/Azure/azure-rest-api-specs`），确认该功能对应的属性/操作是否存在于 **stable** 目录下的 API 版本中，而非仅存在于 **preview** 目录下的版本中。若 Azure 官方文档页面明确标注为 Preview / Public Preview，也视为未 GA。
7. TFVM 仅支持 AzureRM 实现（强制）
	- TFVM 模块的功能实现仅允许使用 `azurerm_*` 资源与其对应 schema。
	- 不在 TFVM 模块内引入 `azapi_resource`、`azapi_update_resource`、`azapi_resource_action` 作为功能实现路径。
	- 当 Azure API 已支持但 AzureRM Provider 未支持时，路由为"等待上游"，而非在模块内绕过 Provider。

## 3. 快速处理流程

对每个功能请求 Issue，按顺序执行：

1. 需求归一化
	- 提炼一句话需求定义：用户想要什么能力、解决什么场景、当前的阻塞点是什么。
2. 属性定位与跨体系映射
	- 在进行代码检查之前，先明确用户请求功能在不同术语体系中的对应关系（Portal 功能名、Azure 文档术语、Azure REST API 属性、AzureRM 属性）。
	- 建议建立映射表：用户原文 → REST API 资源/属性路径 → AzureRM 资源/属性名。
3. 现有支持检查
	- 先检查模块是否已经支持请求功能（基于映射表，同时使用 AzureRM 属性名与 REST API 属性路径进行检索）。
	- 检查层次（从浅到深）：
	  - ① 变量层：`variables.tf` 是否已暴露变量。
	  - ② 资源层（硬编码/默认值）：`main.tf` 等资源文件是否已设置该属性。
	  - ③ 资源层（dynamic 块）：是否由其他变量间接控制。
	- 检查辅助信源：`examples/`、`README.md`、Release/CHANGELOG。
4. 范围判定
	- 判断该功能属于模块职责还是应由用户在模块外处理。
	- 判断是否为 Provider/Azure 平台层面的限制。
5. 实现路径判定（TFVM 固定 AzureRM）
	- 确认该功能是否可在 AzureRM Provider 当前能力内实现。
	- 若 AzureRM Provider 不支持，则进入"等待上游"路径，不设计模块内 AzAPI 补丁方案。
6. 可行性验证
	- 检查 AzureRM Provider 是否已支持对应资源属性或数据源。
	- 检查 Azure REST API 是否已暴露对应能力。
	- 检查功能 GA 状态（若仅 preview，直接路由 F7）。
	- 检查 Terraform 语言层面是否有语法或功能限制。
	- 检查模块现有架构是否能自然扩展容纳该功能。
7. 重复与关联检查
	- 检查是否与 open/closed Issue 或已有 PR 重复。
	- 检查是否有相关上游 Issue（AzureRM Provider 或 Terraform Core）。
8. 兼容性影响评估
	- 评估新增变量/输出对现有用户的影响。
	- 确认新功能是否可纯可选引入（默认值 = 当前行为）。
	- 若不可避免改变现有行为，按破坏性变更流程处理。
9. 实现方案设计
	- 给出一个唯一的默认实现方案。
	- 明确拟新增/修改的变量、资源块、输出。
	- 资源类型必须为 `azurerm_*`，变量 schema 与现有 AzureRM 风格一致。
	- 明确测试策略。
10. 原型验证（建议）
	 - 基于 examples/ 构造原型配置，验证核心路径可行。
	 - 验证新增变量 + 默认值不影响已有示例 plan 结果。
	 - 验证后执行环境清理。
11. 维护者简报输出
	 - 使用第 4 节模板输出可直接决策的信息。

## 4. Agent 输出模板（给维护者）

对每个功能请求 Issue，Agent 输出必须包含：

1. 分类摘要
	- 建议类别：功能请求
	- 置信度：高/中/低
	- 证据摘录：来自 Issue 的关键原文
2. 需求分析
	- 一句话需求定义
	- 用户场景描述
	- 当前阻塞点
	- 属性映射表：（用户原文 → REST API 资源/属性路径 → AzureRM 资源/属性名）
3. 可行性结论
	- Azure 功能 GA 状态：GA/Preview（证据：stable API 版本号或文档链接）
	- 模块实现约束：AzureRM-only（证据：TFVM 规范）
	- Provider 支持状态：已支持/部分支持/未支持
	- API 支持状态：已暴露/未暴露
	- Terraform 语言限制：无/有（说明具体限制）
	- 模块架构兼容性：可自然扩展/需重构/不兼容
	- 证据：引用源代码、文档或 API 定义
4. 兼容性评估
	- 是否可以纯可选方式引入：是/否
	- 对现有用户 terraform plan 的影响：无 drift/可能 drift（说明原因）
	- 是否涉及破坏性变更：否/是（若是，标注 `[BREAKING-CHANGE]`）
5. 实现计划
	- 可行性状态：可实现/需等待上游/不可实现
	- （可实现时）默认方案（必须唯一）：
	  - 实现目标：
	  - 拟新增/修改的变量：（名称、类型、默认值、说明）
	  - 拟新增/修改的资源块：（仅 `azurerm_*`）
	  - 拟新增/修改的输出：
	  - 测试计划：新增测试用例、现有示例回归验证
	  - 风险与缓解：
	- （需等待上游时）：
	  - 阻塞原因：AzureRM Provider Issue 链接 / Terraform Core Issue 链接
	  - 临时替代方案（如有）：模块外自定义实现建议
	- （不可实现时）：
	  - 技术约束说明
	  - 替代方案建议（如有）
6. 维护者行动建议（审批项）
	- 建议决策：批准实现/等待上游/拒绝并说明/要求补充信息
	- 负责人：
	- 时间框：
	- 完成标准：
7. 可直接发送的回复草稿
	- 面向 Issue 提交者的简洁回复，说明当前结论与下一步。

## 5. 可行性验证指南（Agent 执行）

### 5.1 AzureRM Provider 支持检查

1. 在 `terraform-provider-azurerm` 源代码中查找对应资源的 Schema 定义。
2. 确认请求属性是否存在于 Provider Schema 中。
3. 若不存在，检查 Provider 的 open Issue / PR 与路线图信号。
4. 记录 Provider 对应资源文件路径和关键 Schema 定义。

### 5.2 Azure 功能 GA 状态检查（强制，必须先于其他检查）

> **⚠️ 前置阻断检查：若功能为 Preview，直接路由到 F7，不继续后续可行性验证。**

1. 检查 Azure REST API 规范（`github.com/Azure/azure-rest-api-specs`）：
	- 确认属性/操作是否存在于 `stable` API 版本。
	- 若仅存在于 `preview` API 版本，判定为 Preview。
2. 检查 Azure 官方文档（`learn.microsoft.com/en-us/azure/`）：
	- 若页面存在 Preview 标记，判定为 Preview。
3. 记录结论：
	- GA 状态：GA / Preview
	- 证据：stable 版本号、preview 版本号、文档链接
	- 若两者冲突，以 REST API 规范为准

### 5.3 Azure API 支持检查

1. 查阅 Azure REST API 规范，确认请求能力是否在 API 层面已暴露。
2. 若 API 已支持但 AzureRM Provider 未支持，判定为 Provider 层缺口，模块需等待上游支持。
3. 若 API 也未支持，则属于 Azure 平台层限制。

### 5.4 模块架构兼容性检查

1. 检查 `variables.tf`：新功能是否可自然融入现有变量层级。
2. 检查 `main.tf` 与相关资源文件：新增逻辑是否导致复杂度失控。
3. 检查 `outputs.tf`：是否需要新增输出。
4. 检查 `locals.tf`：确认数据转换逻辑影响。

### 5.5 TFVM 实现约束检查（强制）

1. 确认方案仅使用 `azurerm_*` 资源。
2. 确认变量命名与结构遵循 AzureRM 风格（下划线命名、与 AzureRM schema 对齐）。
3. 若方案依赖 AzAPI 才能成立，则该方案在 TFVM 中视为不可接受，应改路由为 F3（等待上游）。

### 5.6 原型验证（建议）

1. 基于最接近的现有 example 创建原型配置，加入新功能变量。
2. 执行 `terraform plan` 验证：
	- 新配置可正常 plan。
	- 默认值配置 plan 无 drift。
3. 如条件允许，执行 `terraform apply` 验证端到端可行性。
4. 验证完成后执行环境清理（优先 `terraform destroy`）。

## 6. 向后兼容性约束（核心）

### 6.1 默认验收标准

新增功能默认必须满足以下标准（除非维护者批准例外）：

1. 纯可选引入
	- 所有新增变量必须有默认值，且默认值等价于"功能未启用"。
2. 计划稳定
	- 对既有稳定配置执行 `terraform plan`，应无 drift。
3. 类型安全
	- 新增变量类型约束尽量精确（优先 `object({...})` 而非 `any`），并附带 validation。
4. 文档同步
	- 新增变量/输出必须补充清晰 description。
	- README 与 examples 需同步更新。

### 6.2 若功能请求隐含破坏性变更（必须高亮）

若请求在技术上无法以纯可选方式引入，Agent 必须在维护者简报中使用醒目标记：

[BREAKING-CHANGE][REQUIRES-MAINTAINER-APPROVAL]

并额外提供：

1. 无法可选引入的技术约束说明。
2. 受影响的现有变量/行为。
3. 迁移指南草案。
4. 版本策略建议（是否需要 major 版本变更）。

## 9. 计划审批门槛（Gate）

在进入代码实现前，Agent 提交的计划至少要满足：

1. 有可行性验证证据（GA 状态确认 + AzureRM Provider 支持检查 + API 支持检查），且功能已确认为 GA。
2. 有明确实现方案，包含变量定义、资源修改和输出变更。
3. 实现方案仅使用 AzureRM 路径（`azurerm_*`），不引入模块内 AzAPI 绕过。
4. 有兼容性评估，确认新增功能对现有用户影响。
5. 有测试计划，覆盖新功能验证和现有示例回归。
6. 若涉及破坏性变更，已提供第 6.2 节全部材料并明确高亮。

## 10. 维护者 KPI（可选但推荐）

按月跟踪：

1. 功能请求 Issue 首次响应时长。
2. F1（可实现且兼容）占比与从受理到计划审批时长。
3. F3（等待上游）平均阻塞时长与重评估频率。
4. 因兼容性风险被转入 F2 的比例。
