# 问题/反馈类 Issue 处理指南（面向维护者）

## 1. 适用范围

本指南定义了当 Issue 被归类为 `问题 / 反馈` 时，Agent 应如何协助维护者处理。

## 2. 简要处理流程

对每个 `问题 / 反馈` 类 Issue，按顺序执行以下步骤：

1. 归纳问题
   用一句话总结维护者需要关注的核心问题。
2. 收集决策上下文
   检查仓库文档/示例和已有模式，看是否已有现成答案。
3. 确定路由
   从 [路由表](#6-问题反馈路由表) 中的 Q1 到 Q6 选择一个路由。
4. 构建维护者简报
   提供包含风险和工作量的紧凑决策报告。
5. 制定下一步行动方案
   明确负责人、时间窗口和完成标志。

### 2.1 特殊路径："如何实现 xxx"

如果提交者在询问实现指导（例如："如何实现 xxx"），使用以下默认行动路径：

1. 构建最小可运行示例
   在 `examples/` 下创建或改写一个示例，反映请求的目标场景。
2. 迭代至可用
   测试和完善示例配置，直到验证出可行的设置。（必须确保幂等性）
3. 记录约束
   记录所有 provider/module 限制、前提假设和必要的前置条件。
4. 回复可执行的指导
   将验证过的配置方案、关键参数和预期结果回复到 Issue 中。

当用户意图是面向实现的时候，应优先采用此路径，而非纯概念性的回答。

## 3. 决策报告模板（Agent → 维护者）

对每个问题/反馈类 Issue，Agent 的输出应包含：

1. 分类摘要
   - 建议类别：
   - 置信度：
   - 理由：
2. 维护者决策信息
   - 用户意图：
   - 仓库中找到的证据：
   - 重复检查：
   - 不处理的风险：
   - 工作量估计：低/中/高
3. 建议路由
   - 路由 ID：Q1/Q2/Q3/Q4/Q5/Q6
   - 选择理由：
4. 可直接发送的维护者回复草稿
5. **代码示例（如涉及配置/代码修改，必须提供）**
   - 如果建议用户修改配置或代码，**必须**在回复草稿中包含一个醒目的、具体的代码示例（用 HCL 代码块展示）。
   - 代码示例应基于仓库 `examples/` 目录中最接近的现有示例改写，而非凭空编造。
   - Agent 必须在给出示例前，阅读相关的 module 源码（变量定义、类型约束、默认值）和 `examples/` 中的用法，**确认示例中的参数名、类型、嵌套结构是真实存在且可用的**。
   - 目标：用户看一眼代码就知道自己应该怎么改，不需要自己去翻文档拼凑。
6. 下一步行动方案（必须）
   - 负责人：
   - 期限：
   - 完成标准：

## 4. 行动方案规则

Agent 必须指定恰好一个默认行动方案，不能给出多个模糊选项。

方案格式：

- 行动：一个明确的步骤。
- 负责人：具体角色。
- 时间框：以工作日为单位的目标。
- 完成标准：客观的完成条件。

实现类问题的默认方案：

- 行动：为请求的场景创建并验证一个最小示例，然后在 Issue 回复中附上经过测试的配置。
- 负责人：维护者或指定的贡献者。
- 时间框：2 个工作日内提供初始示例，后续按需完善。
- 完成标准：示例已验证可用，假设条件已记录，Issue 回复包含可操作的配置。

示例：

- 行动：回复答案并移除 `Needs: Triage` 标签。
- 负责人：值班维护者。
- 时间框：1 个工作日。
- 完成标准：维护者已回复，标签已更新，Issue 状态已解决。

## 5. 维护者 KPI（可选但推荐）

按月跟踪：

1. 问题/反馈类 Issue 的首次维护者响应时间。
2. 路由到 Q2（回答并补充文档）的比例，以减少重复提问。
3. 从问题/反馈转换为 Bug/功能请求的比例。
4. 问题/反馈类 Issue 关闭后的重新打开率。

## 6. 问题/反馈路由表

使用下表进行路由和维护者支持。

| 路由 | 触发条件 | Agent 提供给维护者的信息 | 建议的标签/状态 | 行动方案 |
|---|---|---|---|---|
| Q1: 回答并关闭 | 问题明确，仓库模式/文档中已有答案，无需产品变更。 | 1) 直接回复草稿，2) 仓库文件证据链接，3) 关闭理由 | 保留 `Type: Question/Feedback`，回复后移除 `Needs: Triage` | 负责人：值班维护者。期限：1 个工作日内。完成标准：回复已发布且 Issue 已关闭。 |
| Q2: 回答并补充文档 | 相同问题可能重复出现，或文档/示例虽存在但不够清晰。 | 1) 回复草稿，2) 具体的文档缺口，3) 建议的文档修补范围 | 添加 `Type: Documentation`（或创建关联的文档 Issue/PR），移除 `Needs: Triage` | 负责人：维护者 + 文档贡献者。期限：1 天内回复，3 天内提交文档 PR。完成标准：Issue 关联的文档更新已合并。 |
| Q3: 转为 Bug | 分析过程中发现可复现的错误行为。 | 1) 最小复现摘要，2) 预期行为 vs 实际行为，3) 影响说明 | 重新标记为 `Type: Bug`，如果阻塞则添加 `Needs: Attention` | 负责人：代码维护者。期限：1 天内完成分类决策。完成标准：Bug Issue 已创建/关联，本 Issue 已重定向。 |
| Q4: 转为功能请求 | 用户请求尚不支持的功能，且有明确价值和范围。 | 1) 功能请求摘要，2) provider/AVM 边界检查，3) 兼容性说明 | 重新标记为 `Type: Feature Request` | 负责人：模块维护者。期限：3 天内完成功能分类。完成标准：功能 Issue 已创建/关联，范围已确认。 |
| Q5: 安全升级 | 出现任何漏洞、密钥泄露或滥用风险的迹象。 | 1) 风险摘要（不含敏感信息），2) 即时遏制建议，3) 安全报告指引 | 重新标记为 `Type: Security Bug`，限制公开细节 | 负责人：安全响应人员。期限：立即确认。完成标准：已转入安全处理通道。 |
| Q6: 需要作者反馈 | 因提交者缺少上下文信息，决策被阻塞。 | 1) 缺少的具体字段清单，2) 回复模板，3) 超时建议 | 添加 `Needs: Author Feedback` | 负责人：值班维护者。期限：1 天内追问。完成标准：作者已回复或进入过期策略。 |
